Guía de Conceptos Técnicos

Plataforma Integrada Navistar 2026-2027

Propósito: Este documento explica todos los conceptos técnicos, metodologías y tecnologías mencionadas en el Plan de la Plataforma Integrada, proporcionando el contexto necesario para comprender la propuesta y tomar decisiones informadas.

CONCEPTOS FUNDAMENTALES

Plataforma Integrada

Definición: Una solución tecnológica unificada que combina múltiples sistemas separados (MIOSS y SIV) en una sola aplicación cohesiva, mientras moderniza simultáneamente las interfaces de integración con sistemas corporativos (BaaN → SAP).

Características principales:

  • Unificación de Funcionalidades: Todas las capacidades de ventas en una sola interface
  • Arquitectura Modular: Separación clara de dominios de negocio
  • Integración Nativa: Comunicación directa con SAP sin sistemas intermedios
  • Experiencia Consistente: Misma interface para todos los usuarios

Analogía: Es como convertir una casa con múltiples edificios separados (MIOSS, SIV, interfaces BaaN) en una sola casa moderna con habitaciones bien organizadas y servicios centralizados (electricidad = SAP, plomería = datos).

Beneficios para Navistar:

  • Un solo sistema para mantener en lugar de múltiples
  • Capacitación unificada para usuarios
  • Datos consistentes entre mercados de exportación y nacional
  • Modernización completa de la integración empresarial

ARQUITECTURA Y TECNOLOGÍAS

.NET 8 (Plataforma de Desarrollo Backend)

¿Qué es .NET 8?

.NET 8: Plataforma de desarrollo de Microsoft, versión LTS (Long Term Support) que estará soportada hasta noviembre 2026. Es la evolución moderna de .NET Framework, optimizada para aplicaciones cloud y de alto rendimiento.

¿Por qué .NET 8 para este proyecto?

  • Performance Superior: 25-40% más rápido que versiones anteriores
  • Cloud-Native: Diseñado específicamente para Azure y contenedores
  • Cross-Platform: Funciona en Windows, Linux, macOS
  • Soporte a Largo Plazo: Microsoft garantiza soporte hasta 2026
  • Integración SAP: Conectores nativos y librerías especializadas

Migración desde sistemas actuales: El código existente de MIOSS (C#) y SIV (VB.NET) se puede migrar a .NET 8 manteniendo la lógica de negocio pero aprovechando las mejoras de performance y características modernas.

Ejemplo práctico: Una consulta de precios que actualmente toma 2 segundos en el sistema legacy, tomará menos de 500ms en .NET 8 debido a optimizaciones del compilador y runtime.

Angular 18 (Plataforma de Desarrollo Frontend)

¿Qué es Angular 18?

Angular 18: Framework de desarrollo web de Google para crear aplicaciones de una sola página (SPA). Es la versión más reciente con características modernas como signals, standalone components y server-side rendering.

Ventajas de Angular 18:

  • Signals: Sistema de reactividad más eficiente que reduce re-renderizados innecesarios
  • Standalone Components: Componentes independientes que no requieren módulos
  • Server-Side Rendering (SSR): Carga inicial más rápida
  • Control Flow: Sintaxis más limpia para @if, @for, @switch
  • Bundle Optimization: Aplicaciones hasta 40% más pequeñas

Impacto en usuarios:

  • La aplicación carga en menos de 2 segundos vs 8-10 segundos actuales
  • Navegación instantánea entre módulos sin recargas
  • Funciona sin conexión (PWA - Progressive Web App)
  • Responsive design - funciona igual en desktop, tablet y móvil

Migración desde Angular 6/8: Se aprovechan componentes existentes pero se modernizan para usar las nuevas características, resultando en mejor performance y mantenimiento más fácil.

Progressive Web App (PWA)

¿Qué es una PWA?

Progressive Web App: Aplicación web que usa tecnologías modernas para proporcionar una experiencia similar a una aplicación móvil nativa, pero accesible desde cualquier navegador web.

Características PWA en la plataforma:

  • Instalable: Se puede "instalar" desde el navegador como si fuera una app de escritorio
  • Offline-first: Funciona sin conexión internet para operaciones básicas
  • Push Notifications: Notificaciones como apps móviles nativas
  • Auto-update: Se actualiza automáticamente sin intervención del usuario

Escenario real: Un vendedor en campo puede seguir creando cotizaciones sin conexión internet. Cuando se reconecte, los datos se sincronizan automáticamente con SAP.

Beneficios para usuarios móviles:

  • No necesitan instalar nada desde app stores
  • Funciona en cualquier dispositivo móvil
  • Experiencia consistente entre desktop y móvil
  • Menor consumo de datos que aplicaciones nativas

INTEGRACIÓN SAP

Remediación BaaN → SAP

¿Qué significa "Remediación de Interfaces"?

Remediación: Proceso de reemplazar y modernizar las conexiones e intercambios de datos entre sistemas. En este caso, eliminar la dependencia del sistema BaaN legacy y conectar directamente con SAP.

Estado actual (BaaN):

  • MIOSS y SIV se conectan a BaaN (sistema de los años 90)
  • BaaN actúa como intermediario entre ventas y datos corporativos
  • Interfaces batch (por lotes) que se ejecutan cada 24 horas
  • Datos frecuentemente desactualizados
  • Dificultad para troubleshooting cuando algo falla

Estado objetivo (SAP):

  • Plataforma Integrada se conecta directamente a SAP
  • Interfaces en tiempo real para datos críticos
  • Datos siempre actualizados
  • Mejor trazabilidad y monitoreo
  • Preparado para futuras expansiones

Interfaces a remediar:

  1. Maestro de Clientes: Información de clientes y distribuidores
  2. Lista de Precios: Precios y descuentos por producto
  3. Órdenes de Venta: Pedidos creados en la plataforma → SAP
  4. Inventario: Disponibilidad de productos en tiempo real
  5. Facturación: Generación de facturas en SAP
  6. Cuentas por Cobrar: Estado de pagos de clientes
  7. Reportería Financiera: Dashboards con datos de SAP

Azure Service Bus (Message Bus)

¿Qué es Azure Service Bus?

Azure Service Bus: Servicio de mensajería cloud de Microsoft que permite comunicación confiable entre aplicaciones, incluso cuando una de ellas está temporalmente no disponible.

¿Por qué necesitamos un Message Bus?

  • Reliability: Si SAP está temporalmente no disponible, los mensajes se quedan en cola
  • Scalability: Puede manejar miles de mensajes por segundo
  • Decoupling: La plataforma y SAP no dependen directamente uno del otro
  • Monitoring: Visibilidad completa de todos los mensajes

Flujo típico:

  1. Usuario crea una orden de venta en la plataforma
  2. Sistema envía mensaje a Service Bus: "Nueva orden creada"
  3. Service Bus entrega mensaje a SAP cuando está disponible
  4. SAP procesa la orden y envía confirmación de vuelta
  5. Usuario recibe notificación de que la orden fue procesada

Configuración: Service Bus se configura con diferentes colas para diferentes tipos de mensajes (orders, customers, pricing) con políticas de retry y dead letter queues para manejo de errores.

API Gateway

¿Qué es un API Gateway?

API Gateway: Punto de entrada único que actúa como "portero" para todas las comunicaciones entre la plataforma y sistemas externos como SAP. Maneja autenticación, autorización, rate limiting y logging.

Funciones del API Gateway:

  • Authentication: Verifica que solo usuarios autorizados accedan a SAP
  • Rate Limiting: Controla cuántas llamadas se pueden hacer por minuto
  • Logging: Registra todas las comunicaciones para auditoría
  • Transformation: Convierte datos entre formatos de la plataforma y SAP
  • Caching: Almacena temporalmente respuestas frecuentes

Ejemplo de uso: Cuando un usuario busca precios de productos, el API Gateway:

  1. Verifica que el usuario tiene permisos
  2. Revisa si los precios están en caché (respuesta instantánea)
  3. Si no están en caché, consulta SAP
  4. Transforma la respuesta de SAP al formato esperado
  5. Guarda en caché para futuras consultas
  6. Retorna datos al usuario

CLOUD Y AZURE

Azure App Service

¿Qué es Azure App Service?

Azure App Service: Servicio de Microsoft para hospedar aplicaciones web sin preocuparse por la infraestructura. Microsoft maneja el servidor, sistema operativo, actualizaciones y seguridad.

Beneficios para Navistar:

  • Zero Maintenance: Microsoft maneja todo el mantenimiento de servidores
  • Auto-scaling: Automáticamente agrega más instancias cuando hay más usuarios
  • High Availability: 99.95% uptime garantizado por Microsoft
  • Backup Automático: Respaldos automáticos sin intervención manual
  • SSL Certificates: HTTPS automático y renovación de certificados

Auto-scaling en acción:

  • Horario normal: 2 instancias corriendo (50 usuarios simultáneos)
  • Hora pico: Automáticamente escala a 6 instancias (150 usuarios)
  • Fin de semana: Reduce a 1 instancia (ahorro de costos)
  • Black Friday: Puede escalar a 20+ instancias si es necesario

Costo-beneficio: En lugar de comprar y mantener servidores físicos ($50K inicial + $20K/año mantenimiento), se paga solo por lo que se usa (~$500/mes inicialmente, escalando según necesidad).

Azure SQL Database

¿Qué es Azure SQL Database?

Azure SQL Database: Base de datos SQL Server completamente administrada por Microsoft en la nube. Incluye características avanzadas como backups automáticos, high availability y performance tuning automático.

Características avanzadas:

  • Automatic Backups: Respaldos cada 10 minutos sin configuración
  • Point-in-time Recovery: Restaurar a cualquier momento específico
  • Geo-replication: Copia de la base de datos en otra región geográfica
  • Automatic Tuning: AI de Microsoft optimiza queries automáticamente
  • Threat Detection: Detección automática de intentos de hackeo

Scenario de recuperación: Si por error se eliminan datos importantes a las 2:30 PM, se puede restaurar la base de datos exactamente como estaba a las 2:29 PM, recuperando todo sin pérdida de información.

Para el equipo IT:

  • No más mantenimiento de servidores de base de datos
  • No más preocupación por respaldos
  • Performance monitoring automático
  • Escalamiento automático según demanda

Azure Redis Cache

¿Qué es Redis Cache?

Redis Cache: Sistema de almacenamiento en memoria ultra-rápido que guarda temporalmente datos frecuentemente consultados para evitar ir a la base de datos principal cada vez.

¿Cómo mejora el performance?

  • Velocidad: Acceso a datos en microsegundos vs milisegundos de base de datos
  • Reduced Load: Menos consultas a la base de datos principal
  • User Experience: Respuestas instantáneas para datos comunes
  • Cost Efficiency: Menos recursos necesarios en base de datos

Caso práctico: Lista de precios de productos

  • Primera consulta: Va a SAP, toma 800ms, se guarda en cache
  • Siguientes consultas: Se sirven desde cache en 5ms
  • Actualización: Cada 4 horas se actualiza automáticamente
  • Resultado: Users ven precios instantáneamente

Configuración inteligente: El cache se configura para mantener en memoria los datos más consultados (80/20 rule) y automáticamente libera espacio para nuevos datos cuando es necesario.

METODOLOGÍAS DE DESARROLLO

Sprints de 2 Semanas

¿Qué es un Sprint?

Sprint: Período fijo de tiempo (2 semanas en este proyecto) durante el cual el equipo de desarrollo completa un conjunto específico de tareas y entrega software funcionando.

Estructura de cada Sprint:

  • Día 1: Sprint Planning - decidir qué se va a hacer
  • Días 2-9: Desarrollo y testing
  • Día 10: Sprint Review y Retrospectiva
  • Entregable: Software funcionando al final de cada sprint

Ejemplo Sprint 4:

  • Objetivo: Implementar interface de Maestro de Clientes con SAP
  • Tareas: Desarrollo (6 días), Testing (2 días), Documentación (2 días)
  • Resultado: Clientes de SAP se sincronizan en tiempo real con la plataforma
  • Demo: Usuarios ven la funcionalidad trabajando

Beneficios de sprints cortos:

  • Feedback rápido de usuarios
  • Detección temprana de problemas
  • Flexibilidad para cambiar prioridades
  • Motivación del equipo con logros frecuentes

CI/CD (Continuous Integration/Continuous Deployment)

¿Qué es CI/CD?

CI/CD: Prácticas de desarrollo donde los cambios de código se integran automáticamente, se prueban automáticamente y se despliegan automáticamente a producción.

Continuous Integration (CI):

  • Cada vez que un developer hace commit, se ejecutan pruebas automáticamente
  • Si las pruebas fallan, todo el equipo es notificado inmediatamente
  • Garantiza que el código siempre está en estado funcional

Continuous Deployment (CD):

  • Si todas las pruebas pasan, el código se despliega automáticamente
  • Ambiente de desarrollo se actualiza automáticamente cada día
  • Ambiente de pruebas se actualiza cada sprint
  • Producción se actualiza según calendario aprobado

Pipeline típico:

  1. Developer hace commit a las 10:00 AM
  2. Sistema ejecuta 500+ pruebas automáticamente (10:01 AM)
  3. Si pasan, despliega a ambiente dev (10:05 AM)
  4. Notifica al equipo que nueva versión está disponible
  5. QA puede probar inmediatamente

Herramientas: Azure DevOps maneja todo el pipeline, GitHub almacena el código, y Azure App Service ejecuta los despliegues automáticamente.

Infrastructure as Code (IaC)

¿Qué es Infrastructure as Code?

IaC: Práctica de definir y manejar la infraestructura (servidores, bases de datos, redes) usando código en lugar de configuración manual.

¿Por qué es importante?

  • Reproducible: Misma configuración en dev, test y producción
  • Version Control: Cambios de infraestructura se trackean como código
  • Automated: No errores manuales de configuración
  • Documentation: La infraestructura se auto-documenta

Beneficio práctico: Si el ambiente de producción se corrompe, en 30 minutos se puede recrear exactamente igual usando el código de infraestructura, vs varias horas o días de configuración manual.

Herramienta: Terraform define toda la infraestructura Azure como código, incluyendo App Services, bases de datos, redes, y configuraciones de seguridad.

CONCEPTOS DE INTEGRACIÓN DE DATOS

Entity Framework Core 8

¿Qué es Entity Framework Core?

Entity Framework Core: Herramienta de Microsoft que permite trabajar con bases de datos usando objetos .NET en lugar de escribir SQL directamente. Automáticamente traduce operaciones en objetos a consultas SQL optimizadas.

Ventajas para el desarrollo:

  • Productividad: Developers escriben menos código repetitivo
  • Type Safety: Errores de datos se detectan en compilación, no en runtime
  • Performance: Queries automáticamente optimizadas
  • Migration: Cambios de base de datos se pueden versionar y aplicar automáticamente

Comparación:

SQL Tradicional:

SELECT c.CustomerName, o.OrderDate, o.TotalAmount
FROM Customers c
JOIN Orders o ON c.CustomerId = o.CustomerId
WHERE c.Region = 'LATAM' AND o.OrderDate > '2024-01-01'

Entity Framework:

var orders = customers
    .Where(c => c.Region == "LATAM")
    .SelectMany(c => c.Orders)
    .Where(o => o.OrderDate > DateTime.Parse("2024-01-01"));

Beneficios para mantenimiento:

  • Cambios en estructura de BD se propagan automáticamente
  • Menos bugs relacionados con queries SQL
  • Easier troubleshooting con LINQ debugging
  • Consistencia en acceso a datos across todo el sistema

RESTful APIs

¿Qué son las RESTful APIs?

REST API: Estilo de arquitectura para servicios web que usa HTTP methods (GET, POST, PUT, DELETE) de manera estándar para realizar operaciones sobre recursos.

Principios REST:

  • Stateless: Cada request contiene toda la información necesaria
  • Resource-based: URLs representan recursos (customers, orders, products)
  • HTTP Methods: GET para leer, POST para crear, PUT para actualizar, DELETE para eliminar
  • JSON Format: Datos intercambiados en formato JSON estándar

Ejemplos de endpoints:

  • GET /api/customers - Obtener lista de clientes
  • GET /api/customers/123 - Obtener cliente específico
  • POST /api/orders - Crear nueva orden
  • PUT /api/orders/456 - Actualizar orden existente
  • DELETE /api/products/789 - Eliminar producto

Documentación automática: Usando OpenAPI/Swagger, todas las APIs se documentan automáticamente con ejemplos, parámetros y responses, facilitando integración y troubleshooting.

Real-time vs Batch Processing

¿Cuál es la diferencia?

Real-time Processing: Datos se procesan inmediatamente cuando ocurre un evento. Batch Processing: Datos se acumulan y procesan en grupos a intervalos específicos.

Interfaces Real-time en el proyecto:

  • Maestro de Clientes: Cambios en SAP aparecen inmediatamente en la plataforma
  • Órdenes de Venta: Órdenes creadas se envían inmediatamente a SAP
  • Inventario Crítico: Stock bajo se actualiza inmediatamente

Interfaces Batch en el proyecto:

  • Lista de Precios: Se actualiza diariamente a las 6:00 AM
  • Facturación: Se procesa al final del día
  • Reportería: Se genera mensualmente

Decisión de diseño:

  • Real-time: Para datos que afectan decisiones inmediatas (disponibilidad, precios especiales)
  • Batch: Para volúmenes grandes o procesos que no requieren inmediatez

Balance óptimo: Real-time para experiencia de usuario crítica, batch para eficiencia en procesos masivos, resultando en mejor performance general del sistema.

CONCEPTOS DE CALIDAD Y TESTING

User Acceptance Testing (UAT)

¿Qué es UAT?

User Acceptance Testing: Fase de testing donde usuarios reales prueban el sistema en escenarios reales de trabajo para validar que cumple con sus necesidades de negocio.

Proceso de UAT en el proyecto:

  • Sprint 17-18: Usuarios clave prueban cada módulo
  • Scenarios reales: Casos de uso del día a día
  • Sign-off formal: Usuarios aprueban que la funcionalidad es correcta
  • Issue resolution: Corrección de problemas encontrados

Escenario UAT típico:

  1. Usuario (vendedor real) recibe capacitación de 2 horas
  2. Se le dan tareas específicas: "Crear cotización para cliente XYZ"
  3. Usuario ejecuta el proceso usando datos reales
  4. Se documentan problemas, sugerencias y éxitos
  5. Issues se corrigen antes de go-live

Criterio de éxito: 90% de usuarios deben poder completar sus tareas principales sin ayuda, y 95% deben considerar que el nuevo sistema es igual o mejor que el actual.

Load Testing

¿Qué es Load Testing?

Load Testing: Proceso de simular múltiples usuarios usando el sistema simultáneamente para verificar que puede manejar la carga esperada sin degradación de performance.

Parámetros para Navistar:

  • 100 usuarios concurrentes: Todos los usuarios conectados al mismo tiempo
  • 150 usuarios pico: Carga máxima en horas ocupadas
  • Response time <200ms: Tiempo máximo aceptable para respuestas
  • Zero downtime: Sistema debe mantener disponibilidad

Scenario de prueba:

  • 50 usuarios creando cotizaciones simultáneamente
  • 30 usuarios consultando inventario
  • 20 usuarios generando reportes
  • Sistema debe responder en menos de 200ms para cada operación

Herramientas: Azure Load Testing simula carga real y proporciona métricas detalladas de response times, throughput y resource utilization.

Security Testing

¿Qué incluye Security Testing?

Security Testing: Conjunto de pruebas diseñadas para identificar vulnerabilidades de seguridad y asegurar que el sistema protege adecuadamente datos y funcionalidades.

Áreas de testing de seguridad:

  • Authentication Testing: Verificar que solo usuarios autorizados pueden acceder
  • Authorization Testing: Verificar que usuarios solo acceden a lo que les corresponde
  • Data Encryption: Verificar que datos sensibles están encriptados
  • SQL Injection: Intentos de hackear la base de datos
  • Cross-site Scripting: Intentos de inyectar código malicioso

Pruebas específicas:

  • Intentar acceder con passwords débiles o comunes
  • Intentar acceder a datos de otros usuarios
  • Verificar que passwords no se almacenan en texto plano
  • Verificar que sesiones expiran automáticamente

Certificaciones obtenidas: El sistema cumplirá con estándares OWASP Top 10, ISO 27001 requirements y regulaciones locales de protección de datos.

CONCEPTOS DE MONITOREO Y OBSERVABILIDAD

Application Insights

¿Qué es Application Insights?

Application Insights: Servicio de Microsoft que automáticamente recolecta telemetría detallada de aplicaciones, incluyendo performance, errores, user behavior y dependencias externas.

Métricas automáticas que recolecta:

  • Response Times: Tiempo que toma cada request
  • Error Rates: Porcentaje de requests que fallan
  • Dependency Calls: Tiempo de respuesta de SAP, base de datos, etc.
  • User Sessions: Cuántos usuarios están conectados
  • Custom Events: Eventos específicos del negocio

Dashboard en tiempo real muestra:

  • "Orden creada" - 45 veces en la última hora
  • "Error conectando SAP" - 2 veces (99.5% success rate)
  • "Response time promedio" - 150ms (dentro del objetivo)
  • "Usuarios activos" - 67 de 100 total

Alertas automáticas: Sistema envía emails/SMS cuando response time > 500ms, error rate > 2%, o SAP no responde por más de 5 minutos.

Health Checks

¿Qué son Health Checks?

Health Checks: Verificaciones automáticas que el sistema ejecuta cada minuto para confirmar que todos los componentes están funcionando correctamente.

Componentes monitoreados:

  • Database Connection: ¿Se puede conectar a Azure SQL?
  • SAP Connectivity: ¿Responde SAP a un ping?
  • Redis Cache: ¿Está disponible el cache?
  • File Storage: ¿Se pueden leer/escribir archivos?
  • External APIs: ¿Responden servicios externos?

Health check endpoint responde:

{
  "status": "Healthy",
  "totalDuration": "00:00:00.1230000",
  "entries": {
    "database": { "status": "Healthy" },
    "sap": { "status": "Healthy" },
    "redis": { "status": "Degraded", "description": "High latency" }
  }
}

Beneficio operacional: IT team puede identificar y resolver problemas antes de que los usuarios los noten, mejorando significativamente la experiencia y uptime.
